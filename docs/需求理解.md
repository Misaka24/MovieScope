# MovieScope 电影搜索与评分网站需求分析

> 当前实现说明（2026-07-12）：TMDB 是主体数据源；IMDb 数据统一由 Just One API 提供；豆瓣只接入评分、长评和近期热门，短评已取消。详情页当前保留站内星级评分与站内评价，已取消“想看/在看/看过/收藏”和观看平台展示。本文后续早期需求记录用于追踪演进；冲突处以《项目与架构说明》和当前源码为准。

## 1. 需求概述

MovieScope 是一个参考豆瓣电影与 IMDb 的影视信息、发现、记录和评价平台。平台面向希望搜索影视作品、查看聚合资料、发现新内容并长期管理个人观影记录的用户。

系统以 Vue 3 + Vite 构建 Web 前端，通过后端统一接入多个真实影视数据 API，并用数据库保存账号、观影状态、浏览与搜索历史、收藏、评分和短评。前端视觉稿与主要设计方向由需求方提供；开发实现需保持设计意图，同时针对真实 API 的字段差异、图片比例、文本长度、数据缺失、限流和失败状态进行适配。

本阶段定义“做什么”和验收边界，不锁定具体后端框架、数据库产品、API 供应商及部署架构，这些内容留待技术方案设计阶段确认。

## 2. 产品目标

1. 提供统一、快速、可信的影视搜索入口。
2. 聚合多个外部真实数据源，形成较完整的电影、剧集、人物和评分信息展示。
3. 打通“发现内容—查看详情—记录状态—评分评价—个人管理”的用户闭环。
4. 持久化用户观影与评价数据，使其在刷新、重启和重新登录后仍可恢复。
5. 在外部数据缺失、冲突、限流或暂时不可用时，提供可理解、可恢复的体验。

## 3. 目标用户与核心场景

### 3.1 用户角色

- 游客：搜索和浏览公开影视资料，不保存个人数据。
- 注册用户：维护观影状态、收藏、评分、短评与个人历史。
- 影视爱好者：查看较完整的主创、剧情、图片、预告和多来源评分。
- 管理员：处置异常内容和违规评价；完整运营后台不属于首期 P0。

### 3.2 核心场景

- 按片名、别名、关键词或人物找到目标作品。
- 按热门、趋势、类型、地区、年份等条件发现内容。
- 查看剧情、演职员、上映信息、媒体资源、站内及外部评分。
- 将作品标记为“想看”“在看”“看过”，进行收藏、评分和短评。
- 在个人主页查看、筛选、修改或删除个人记录。
- 查看近期浏览与搜索历史，重新进入曾访问的内容。

# 外部 API 缓存存储约束补充

1. TMDB、豆瓣和 IMDb 的接口响应缓存必须持久化写入 SQL 数据库，禁止使用进程内 Map、对象、LRU 或仅存在于运行内存中的缓存。
2. 本地开发允许使用 PGlite 作为持久化 SQL 数据库；生产环境使用 PostgreSQL，两者须共用同一套表结构和参数化 SQL 数据访问逻辑。
3. 缓存记录必须包含数据源、缓存键、响应内容、抓取时间和过期时间；数据源同步状态、耗时与错误必须可追踪，供后续管理员后台展示。
4. 应用进程或机器重启后，未过期缓存仍应可读取，不得因进程重启丢失。
5. 浏览器只访问 MovieScope 后端，不得直接访问第三方密钥接口或持有 TMDB、豆瓣、IMDb 的访问令牌。

## 5. 页面与信息架构

### 5.1 全局框架

- 导航包含主页、搜索、探索、个人主页入口。
- 提供全局搜索、登录/注册、用户菜单和退出入口。
- 提供加载、空结果、错误、网络异常、限流及操作反馈。
- 支持常见桌面端和移动端布局。

### 5.2 主页

- 展示热门、近期趋势、正在上映、即将上映或高分内容。
- 至少包含三个由真实 API 或后端缓存驱动的内容分区。
- 支持从分区进入详情页或对应探索结果。
- 登录用户可看到基于浏览或观影历史生成的继续探索入口；首期不要求复杂推荐算法。

### 5.3 搜索页

- 支持片名、原名、别名、关键词和演职员搜索。
- 结果至少区分影视与人物，或提供明确类型标识。
- 支持电影/剧集、年份、类型等基础筛选，具体项目按数据源能力调整。
- 支持分页或连续加载，并保持关键词和筛选状态。
- 支持搜索建议、无结果提示和关键词调整入口。
- 登录用户的有效搜索可写入搜索历史，并允许删除或清空。

### 5.4 探索页

- 支持热门、趋势、高分、上映状态等浏览维度。
- 支持类型、地区/语言、年份、媒体类型等组合筛选。
- 支持按热度、评分、上映时间等可用字段排序。
- 筛选条件应写入 URL 或可恢复状态，便于刷新和分享。
- 数据源不支持的条件应隐藏、禁用或调整，不得伪造筛选结果。

### 5.5 影视详情页

- 展示海报、背景图、片名、原名、年份、类型、片长、地区、语言和上映状态。
- 展示剧情、导演、编剧、主演和角色信息。
- 展示站内评分统计，并与可合法获取的外部评分明确区分来源。
- 展示剧照、海报、预告片等可用媒体资源。
- 展示相关推荐、相似作品或系列内容。
- 登录用户可设置状态、收藏、评分和短评。
- 展示公开短评列表，支持基础排序或分页。
- 缺失字段应隐藏或使用友好占位，不得暴露技术空值。

### 5.6 人物详情页

- 人物搜索结果可进入人物详情页，首期最低可采用独立结果视图。
- 展示姓名、头像、简介、职业和代表作品。
- 可从人物作品列表进入影视详情。

### 5.7 登录与注册页

- 支持注册、登录、退出、登录状态保持和过期处理。
- 提供字段校验、重复账号、错误凭证和密码要求提示。
- 未登录用户触发个人操作时提示登录；成功后尽量返回原页面或继续原操作。

### 5.8 个人主页

- 展示头像、昵称、简介、注册时间等资料。
- 展示想看、在看、看过和收藏清单。
- 展示评分、短评、浏览历史和搜索历史。
- 支持筛选、修改、删除单项或清空适用的历史记录。
- 默认仅本人可见私有历史；公开主页及细粒度隐私属于增强能力。

### 5.9 通用状态页

- 404：路由或资源不存在。
- 服务异常：后端或关键数据源不可用。
- 部分数据：作品存在，但部分资料无法取得。
- 鉴权失效：提示重新登录，并保留可恢复页面上下文。

## 7. 数据需求

### 7.1 外部数据范围

外部数据应覆盖以下类别中的大部分：影视基础信息和 ID、多语言标题、简介、类型、发行信息、海报与剧照、演职员与人物资料、热门/趋势/上映列表、预告片链接、可合法展示的外部评分。

外部影视数据来源限定为 TMDB、豆瓣电影 API 和 IMDb API。三类 API 的具体版本、授权条款、额度和调用方式依据项目实际获得的对应 API 文档与密钥确认，不接入其他影视数据 API。

### 7.2 聚合规则

- 保存每个来源的原始 ID，建立站内影视实体与外部实体映射。
- 优先使用稳定外部 ID 关联；标题、年份等仅作为辅助匹配条件。
- 字段冲突按预定义来源优先级处理，并保留来源和最近同步时间。
- 站内评分独立计算，不能与外部评分直接平均。
- 外部作品暂时不可用时，用户历史仍应保留并显示内容不可用状态。

### 7.3 站内核心实体

- 用户：账号、凭证摘要、昵称、头像、简介、状态和时间信息。
- 影视映射：站内 ID、媒体类型、外部来源 ID、缓存摘要和同步时间。
- 观影记录：用户、作品、状态、创建与更新时间。
- 用户评分：用户、作品、分值、创建与更新时间。
- 用户短评：用户、作品、正文、公开/审核状态和时间。
- 收藏记录：用户、作品、收藏时间。
- 浏览历史：用户、作品、最近访问时间、必要的访问次数。
- 搜索历史：用户、关键词、筛选摘要和时间。
- 会话或令牌记录：依据最终鉴权方案确定。

## 8. 关键业务规则

1. 同一用户对同一作品最多有一个当前观影状态、一条当前评分和一条首期短评。
2. 全站采用统一评分制式，建议在 10 分制或 5 星制中二选一；数据库只保存一种标准值。
3. 评分变更后，站内聚合统计应同步更新。
4. 标记“看过”不强制要求评分或短评。
5. 游客可浏览公开内容，但个人数据写入必须登录。
6. 浏览历史仅记录成功访问的作品详情；重复访问更新最近时间而非无限新增。
7. 删除观影状态不自动删除评分或短评，除非产品交互明确要求并确认。
8. 外部 API 密钥不得暴露在浏览器端，受保护请求由后端代理。
9. 设计稿与真实数据冲突时，优先保证内容可读、操作可达和状态完整。

## 9. 前后端交互要求

- 前端通过统一站内 API 获取聚合数据，不直接依赖多个供应商响应结构。
- 后端提供标准化模型，并明确哪些字段可空。
- 用户写操作必须鉴权，并验证数据所有权。
- 创建、修改、删除成功后返回最新资源状态，避免无依据的前端假更新。
- 列表接口使用一致分页结构，返回总量、页码或游标及是否还有更多。
- 错误至少区分参数错误、未登录、无权限、不存在、冲突、限流和服务异常。
- 评分、短评和状态操作需要防重复提交或幂等保护。
- 退出登录后清除本地敏感状态，可保留非敏感筛选偏好。

## 10. 非功能需求

### 10.1 性能

- 首屏优先展示骨架和关键内容，非关键图片与分区延迟加载。
- 正常网络下，已缓存核心读取接口目标响应时间不高于 500 ms；实时聚合接口目标不高于 3 s。
- 搜索建议使用防抖；图片使用合适尺寸、懒加载、占位和失败回退。
- 外部调用具备超时、有限重试、缓存和限流保护。

### 10.2 安全与隐私

- 密码使用可靠单向哈希，禁止明文存储和日志输出。
- API 密钥、数据库凭证和鉴权秘密不得进入前端产物或公开仓库。
- 服务端验证所有输入，并防范注入、XSS、越权及恶意超长内容。
- 私有历史默认仅本人可访问，权限不能只依赖前端隐藏。
- 登录、注册、搜索和评价接口具备合理频率限制。
- 生产环境使用 HTTPS 和安全的会话/Cookie 策略。

### 10.3 可用性与可靠性

- 单个非关键数据源失败不能导致整页不可用。
- 外部服务超时后返回缓存、部分数据或明确错误，不无限等待。
- 所有异步页面具备加载、成功、空、部分内容和失败状态。
- 写操作失败时不得永久显示为成功。
- 关键用户数据写入具备事务一致性或等价保护。

### 10.4 兼容性与可维护性

- 支持主流桌面和移动浏览器近期稳定版本。
- 至少覆盖约 360 px 移动端、平板和 1280 px 以上桌面布局。
- 超长标题、多语言、缺图和图片比例差异不得造成重叠或阻断操作。
- 记录外部 API 调用、缓存命中和同步失败日志，但不记录敏感凭证。
- 供应商适配层可替换，业务页面不得直接绑定供应商字段。
- 核心规则具备自动化测试，关键流程具备端到端验证。

## 11. 边界与约束

### 11.1 已明确约束

- 前端以 Vue 3 + Vite 为主。
- 视觉设计主要由需求方提供，实现方负责对齐、组件化和真实数据适配。
- 影视数据来自多个真实 API，不以长期静态假数据作为正式来源。
- 用户账号、状态、历史、评分和评价必须数据库持久化。
- 必须包含主页、搜索页、探索页、个人主页和影视详情页。

### 11.2 首期不包含

- 影视在线播放、盗版资源聚合或视频文件托管。
- 对豆瓣或 IMDb 的数据、视觉和功能进行一比一复制。
- 未经许可抓取、复制第三方受版权保护的评论。
- 完整论坛、即时聊天、商业广告、会员付费、票务和电商交易。
- 机器学习级推荐、原生 App 和完整多语言系统。

### 11.3 主要风险

- API 授权、额度、字段完整度和中国大陆网络可用性影响供应商选择。
- 部分来源限制缓存或要求署名，展示必须遵守许可协议。
- 多源匹配可能出现重名、年份偏差、地区版本差异和重复条目。
- 图片可能失效、比例不一或缺少中文资源，需要回退方案。
- 纯 SPA 的搜索引擎收录和社交预览能力可能受限。

## 12. 验收标准

### 12.1 数据聚合

- Given 至少配置两个真实 API，When 打开主页、搜索或详情，Then 展示真实接口数据，并能识别关键数据来源。
- Given 一个非关键来源失败，When 其他来源或缓存可用，Then 页面仍展示可用内容并提示缺失，不整体崩溃。
- Given 两个来源返回同一作品，When 后端聚合，Then 按 ID 或既定规则映射为一个站内实体。

### 12.2 主页、搜索与探索

- Given 外部数据正常，When 进入主页，Then 至少展示三个真实数据分区，作品可进入详情。
- Given 用户提交有效关键词，When 搜索完成，Then 展示相关结果并支持筛选和翻页/加载更多。
- Given 搜索无结果，When 请求完成，Then 显示明确空状态并允许快速修改关键词。
- Given 用户组合探索条件，When 刷新页面，Then 条件与结果可恢复。

### 12.3 详情页

- Given 打开有效作品，When 加载完成，Then 展示基础资料、剧情、演职员、评分和媒体中的可用字段。
- Given 海报、简介或评分缺失，When 页面渲染，Then 使用友好回退或隐藏模块，不出现技术错误或布局重叠。
- Given 存在相关推荐，When 点击推荐项，Then 进入正确作品详情。

### 12.4 账号与权限

- Given 合法且未注册的信息，When 注册成功，Then 数据库创建用户且该用户可登录。
- Given 错误凭证，When 登录失败，Then 返回不泄露敏感细节的错误且不创建会话。
- Given 会话过期，When 执行受保护操作，Then 提示重新登录且不误报成功。
- Given 用户修改他人评价，When 请求到达后端，Then 返回无权限且不改变数据。

### 12.5 用户数据闭环

- Given 用户已登录，When 设置观影状态，Then 数据库保存唯一当前状态，刷新和重新登录后保持一致。
- Given 用户已有评分，When 修改评分，Then 更新原记录而非新增重复记录，聚合统计随之变化。
- Given 用户保存短评，When 成功，Then 详情页与个人主页均可查看；删除后两处均不再显示。
- Given 用户收藏作品，When 刷新或重新登录，Then 收藏状态与个人列表一致。
- Given 用户访问有效详情，When 查看浏览历史，Then 同一作品合并为一个条目并更新最近时间。
- Given 用户清空历史，When 操作成功并刷新，Then 已清空数据不再出现。

### 12.6 设计与安全

- Given 已确认设计稿，When 前端完成，Then 结构、色彩、字体层级、间距和交互状态保持一致。
- Given API 返回超长文本、缺图或不同比例图片，When 在移动与桌面端展示，Then 无重叠且关键操作可达。
- Given 检查前端构建产物和公开仓库，Then 不包含私密 API 密钥、数据库密码或用户密码。
- Given 服务重启且数据库保留，When 用户重新登录，Then 个人状态、收藏、评分、短评和历史可恢复。

## 13. MVP 完成定义

1. 主页、搜索、探索、详情、登录/注册和个人主页形成完整导航。
2. 至少两个真实数据源经后端接入，关键页面不依赖静态假数据。
3. 用户可注册、登录、退出，并在重新登录后恢复个人数据。
4. 用户可管理状态、收藏、评分、短评、浏览和搜索历史。
5. 核心页面具备加载、空、错误和部分数据状态，适配移动与桌面端。
6. 密钥和凭证得到服务端保护，基本越权和输入安全检查通过。
7. 核心接口和关键用户流程具备自动化测试或可重复验收脚本。

## 14. 假设与待确认事项

1. 后端技术栈：Node.js/TypeScript、Java、Python 或其他方案。
2. 数据库、部署平台、服务器区域、预算和预期并发规模。
3. 最终 API 组合、付费额度接受程度及中国大陆直连要求。
4. 首期是否完整支持电影与剧集，还是电影优先。
5. 评分采用 10 分制还是 5 星制，是否允许半星。
6. 每个用户每部作品是否只允许一条短评，是否需要长评。
7. 个人主页默认私有、部分公开，还是允许用户配置隐私。
8. 登录采用用户名、邮箱、手机号或第三方方式的具体范围。
9. 设计稿工具、页面覆盖、断点、组件状态和交付节奏。
10. 首期是否需要可视化管理后台。
11. 目标运营地区及备案、隐私政策、Cookie 和内容审核要求。
12. 项目定位是课程/作品集、公开非商业产品，还是正式商业运营。

## 15. 需求结论

当前需求可确定为一个包含真实多源影视数据聚合、用户账号体系、搜索与发现、影视详情和个人观影数据管理的完整前后端 Web 应用。首期应优先保证真实数据链路、用户数据闭环和关键页面完整性；复杂社交、智能推荐、在线播放和商业能力不进入 P0。

下一阶段应开展技术方案设计，重点确认系统架构、外部 API 组合与字段映射、数据库模型、鉴权方式、缓存策略、接口契约、前端路由与组件边界、测试策略和部署方案。

# 需求补充：真实 API、双端产品与实施顺序

在原需求基础上增加以下确定项：

4. 前端 Mock 必须遵循统一接口契约，后端实现同一契约；不得将假数据直接散落在页面组件中。
5. 最终真实外部数据源仅包括 TMDB、豆瓣电影 API 和 IMDb API：TMDB 负责主体目录，豆瓣负责中文口碑，IMDb 负责国际化与深度信息。
6. 豆瓣电影 API 和 IMDb API 均属于项目规定的数据源；具体 Base URL、版本和鉴权方式以需求方后续提供或确认的实际接口文档为准。
7. 所有外部 API 请求必须由后端代理，统一执行密钥保护、缓存、限流、超时、重试、映射和错误降级。
8. 项目分为四个主要阶段：前端双端与契约定型、后端基础与用户闭环、真实 API 聚合与全站联调、质量收口与部署。阶段不再按单个页面或单个接口细碎拆分。

10. 站内用户评分和短评独立保存，不与外部评分直接平均；外部评分必须显示来源及同步时间。

完整技术范围、接口、数据模型和阶段验收以 `docs/方案设计.md` 为准。
