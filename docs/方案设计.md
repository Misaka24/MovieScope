# MovieScope 技术方案设计

> 当前基线：2026-07-12。本方案对应当前仓库的真实实现。需求边界见《需求理解》，面向工程阅读的完整目录和接口索引见《项目与架构说明》。

## 1. 设计目标

系统需要在不暴露外部密钥的前提下，将 TMDB、Just One IMDb、Just One 豆瓣、Wikidata 和站内 MySQL 数据统一为稳定的 MovieScope API；同时控制付费调用、支持上游故障降级、实现用户与管理员业务，并保持前端主题与交互一致。

设计重点：

- TMDB 免费数据优先，Just One 只承担明确需要的深度数据。
- 外部 Provider 与页面解耦，后端统一字段。
- MySQL 同时承担业务存储、Provider 配置和持久化缓存。
- 使用按上映时间动态变化的 TTL，而不是全局固定缓存。
- IMDb→豆瓣映射必须精确验证，不能仅依赖标题模糊匹配。
- 详情页的评论、奖项、票房和花絮能够容纳长文本与缺失字段。

## 2. 架构

```text
Vue 3 SPA
  │ HTTP /api + HttpOnly Cookie
  ▼
Node.js HTTP Server
  ├─ Catalog/Home aggregation
  ├─ User/Auth/Profile
  ├─ Admin/RBAC/Moderation
  ├─ Provider adapters
  ├─ Cache policy + SQL cache
  └─ Identity mapping
  │
  ├──────── TMDB API
  ├──────── Just One IMDb API
  ├──────── Just One Douban API
  ├──────── Wikidata SPARQL
  ▼
MySQL 8
```

前端只依赖站内契约。`server/providers.mjs` 是外部访问边界；`server/cache.mjs` 在 Provider 外围完成 SQL 缓存；领域服务负责聚合和字段转换；`server/server.mjs` 暴露 HTTP 路由。

## 3. 技术栈与选型

### 前端

- Vue 3.5：Composition API 和响应式状态。
- TypeScript 5.9：前后端数据契约和页面类型检查。
- Vue Router 4：SPA 路由与懒加载。
- Vite 8：开发服务器、代理、HMR 和生产构建。
- Tailwind CSS 3 + 原生 CSS：布局、主题、动画和复杂页面样式。
- Material Symbols、Inter、JetBrains Mono：图标和字体。

### 后端

- Node.js ESM：运行环境。
- 原生 `node:http`：HTTP 服务和路由，避免额外框架层。
- 原生 Fetch、AbortSignal timeout：外部请求与超时。
- `mysql2/promise`：MySQL 连接池和异步查询。
- `node:crypto` scrypt：密码哈希、会话 Token 哈希。
- Node Test Runner：单元测试。

### 基础设施

- MySQL 8.4、utf8mb4。
- Dockerfile.api、Dockerfile.web。
- Docker Compose：MySQL、API、Web 三服务。
- Nginx：SPA 静态托管与 `/api` 反向代理。
- GitHub Actions：检查、镜像构建、GHCR 推送、SSH 部署和健康检查。

## 4. 前端设计

### 路由

`src/router.ts` 定义首页、搜索、探索、影视详情、完整演职员、完整剧照、影人、榜单、影人列表、通知、平台、新闻、认证、个人主页、公开主页和后台共 16 类页面路由。页面组件使用动态 import 分包。

### 数据访问

- `src/services/api-client.ts`：统一 fetch、错误格式、Cookie 和 query 参数。
- `catalog-api.ts`：目录、详情、评论、榜单、新闻和影人。
- `home-api.ts`：首页聚合。
- `user-api.ts`：认证、用户、互动、历史和后台接口。
- `src/types`：API 返回类型；外部原始字段只在确有必要时通过 `rawData` 保留。

### 状态与组件

- `useAsyncData`：异步 loading/error/retry。
- `useAuth`：当前用户和会话。
- `useHomeData`、`useHeroCarousel`：首页数据和轮播。
- CatalogPosterCard/Grid、PaginationBar、PageState：列表页复用。
- GlobalHeader/Footer：全站导航、搜索和账号入口。

### 详情页评论逻辑

评论平台状态为 `tmdb | imdb | douban`，默认 TMDB。统一的计算属性完成剧透过滤、热度/时间排序和五条切片；点击查看更多只增加可见数量，直到后端已获取数据全部显示。豆瓣长评展开时才调用详情接口；TMDB 和 IMDb 正文直接使用已返回内容，避免错误付费调用。

## 5. 后端设计

### HTTP 层

`server/server.mjs` 实现：

- JSON 响应统一为 `{ data, error }`。
- 请求体最大 1 MiB。
- CORS 只允许 `WEB_ORIGIN`。
- Cookie 解析与 HttpOnly 会话。
- 维护模式、注册开关和管理员绕过。
- 404、业务错误和未捕获错误边界。
- SIGINT/SIGTERM 优雅关闭。

### 领域服务

- `home-service.mjs`：并行获取首页来源，单源失败不拖垮整页。
- `catalog-service.mjs`：搜索、发现、详情、榜单、影人、外部评价、票房和新闻。
- `douban-service.mjs`：映射、条目、长评分页和长评详情。
- `user-service.mjs`：账号、会话、密码、站内评价和历史。
- `admin-service.mjs`：RBAC、用户、审核、Provider、缓存、设置、统计和日志。
- `runtime-config.mjs`：从数据库读取 Provider 与站点运行配置。

## 6. 外部接口设计

### TMDB

实际使用的接口族：

- `/search/multi|movie|tv|person`：搜索。
- `/discover/movie|tv`：探索筛选。
- `/genre/movie/list`、`/genre/tv/list`：类型。
- `/movie/now_playing`、`popular`、`top_rated`、`upcoming`。
- `/tv/popular`、`top_rated`、`airing_today`。
- `/trending/*/day|week`：趋势。
- `/{movie|tv}/{id}` + `append_to_response`：详情、credits、images、videos、external_ids、release/content ratings、recommendations、similar、keywords、translations。
- `/{movie|tv}/{id}/reviews`：TMDB 用户评论全部分页。
- `/person/{id}`、`/person/popular`：影人。
- `/find/{imdbId}`：IMDb ID 解析与中文补全。
- `/watch/providers/movie|tv`：平台目录。
- `/configuration/languages`：语言配置。

TMDB 评论第一页返回 `total_pages`，后端最多接受 500 页，每批并行 10 页；每页独立缓存 6 小时，合并后按 ID 去重。头像路径兼容 TMDB 图片路径和外部 Gravatar URL。

### Just One IMDb

- `/api/imdb/title-redux-overview-query/v1`
- `/api/imdb/title-user-reviews-summary-query/v1`
- `/api/imdb/title-critics-review-summary-query/v1`
- `/api/imdb/title-awards-summary-query/v1`
- `/api/imdb/title-did-you-know-query/v1`
- `/api/imdb/title-box-office-summary/v1`
- `/api/imdb/title-chart-rankings/v1`，参数 `TOP_250` 或 `TOP_250_TV`
- `/api/imdb/news-by-category-query/v1`

所有 IMDb ID 来自 TMDB `external_ids` 或榜单本身，不以标题猜测。Redux 负责 IMDb 评分；奖项、票房和“你知道吗”按独立结构转换。专业评论与用户评价保留尽可能多的接口字段。

### Just One 豆瓣

- `/api/douban/get-subject-detail/v1`
- `/api/douban/get-movie-reviews/v1`
- `/api/douban/get-movie-review-detail/v1`
- `/api/douban/get-recent-hot-movie/v1`
- `/api/douban/get-recent-hot-tv/v1`

不调用 `/get-movie-comments/v1`。长评列表默认同步配置页数并去重，前五篇预取完整正文；其余在用户展开时补拉，正文缓存 10 年。

### Wikidata 映射

映射步骤：

1. 以 IMDb ID 查询 Wikidata SPARQL 中的豆瓣标识。
2. 请求候选豆瓣条目详情。
3. 比较详情中的 IMDb ID，只有完全一致才保存映射。
4. 若 Wikidata 失败，以中文标题、原始标题和年份搜索候选。
5. 对搜索候选重复精确验证。
6. 成功保存 180 天，失败记录 7 天，避免每次页面访问重复搜索。

## 7. 缓存方案

`cachedSql` 的缓存键包含 Provider、路径和规范化参数。正常查询只读取未过期数据；加载时使用同 key 的进程内 Promise 锁避免并发击穿；成功后写入 payload、fetched_at 和 expires_at；失败时查询 365 天内最近成功数据作为 stale。

| 数据 | TTL |
| --- | --- |
| 通用单片数据，上映 ≤14 天 | 1 天 |
| 上映 14–60 天 | 7 天 |
| 上映 60–90 天 | 30 天 |
| 上映 >90 天或日期未知 | 60 天 |
| 奖项，上映 ≤180 天 / >180 天 | 7 天 / 45 天 |
| “你知道吗”，上映 ≤30 天 / >30 天 | 7 天 / 45 天 |
| IMDb Top 250 电影/剧集 | 7 天 |
| 豆瓣近期热门 | 3 天 |
| IMDb 新闻 | 1 天 |
| TMDB 评论页 | 6 小时 |
| 豆瓣完整长评 | 10 年 |
| 映射成功 / 失败 | 180 天 / 7 天 |

Just One Provider 不设置全局最低 TTL，业务错误不额外重试。TMDB 仍可使用较短 TTL，因为免费且是主体来源。

## 8. 数据库设计

### 缓存与同步

- `api_cache`：缓存键、Provider、JSON payload、抓取和过期时间。
- `provider_sync_logs`：调用结果、延迟和错误记录。

### 用户与认证

- `users`：账号、角色、状态、资料和可见性。
- `user_sessions`：Token 哈希、设备信息和过期/撤销时间。
- `auth_login_attempts`：登录限流。
- `user_media_entries`：评分、评价、剧透、兼容观影状态和影视快照。
- `user_browse_history`、`user_search_history`：历史。
- `user_media_events`：影视行为事件与分析。

### 管理系统

- `admin_audit_logs`：管理员审计。
- `admin_groups`、`admin_permissions`、`admin_group_permissions`：权限模型。
- `user_admin_groups`：用户组关系。
- `user_permission_overrides`：用户级 allow/deny。
- `admin_user_notes`：管理员备注。
- `api_provider_configs`：Provider 地址、启用、超时和说明。
- `site_settings`：注册、维护、公告等配置。

### 外部身份

- `media_identity_mappings`：TMDB、IMDb、豆瓣 ID、状态、验证和过期时间。

迁移使用 `CREATE TABLE IF NOT EXISTS` 和条件 ALTER，API 启动时按文件名顺序执行，适合新环境初始化和已有环境升级。

后台 Provider 目录只使用 `tmdb` 和 `justone` 两个系统 key。`justone` 对应同一个 Just One API 服务，统一承载 IMDb 与豆瓣接口。`006_provider_catalog_sync.mysql.sql` 会将历史 `imdb` 配置、缓存和日志迁移到 `justone` 并删除 `imdbapi-dev`，因此新旧数据库最终显示一致。

## 9. 认证与权限

密码长度校验后使用随机盐 scrypt。登录成功生成随机 Token，数据库只保存 SHA-256 哈希，浏览器获得 HttpOnly Cookie。管理员权限由角色、组权限和用户级覆盖合并；显式 deny 优先于 allow。每个后台路由都声明所需权限，不能只依赖前端隐藏按钮。

## 10. 部署方案

### 本地

`.env` 配置 TMDB、Just One 和 MySQL。`npm run dev:all` 启动 API 与 Vite；Vite 将 `/api` 代理到 8787。

### 生产

- `mysql`：MySQL 8.4，持久卷 `mysql_data`。
- `api`：Node 镜像，等待 MySQL healthy，暴露内部 8787。
- `web`：Nginx 镜像，监听宿主 `127.0.0.1:8080`。
- 宿主 Nginx 使用 `deploy/nginx-host.conf.example` 代理公网域名。
- GitHub Actions 执行 `npm ci`、检查、构建镜像、推送 GHCR、SSH 备份数据库、拉取镜像、重启和健康检查。

## 11. 测试与可观测性

- `vue-tsc -b`：模板与类型。
- `node --test server/*.test.mjs`：权限、缓存 TTL、IMDb 字段、首页去重、中文标题和密码。
- `vite build`：生产构建。
- Provider 同步日志记录成功、失败和耗时。
- 后台展示 Provider、缓存和审计信息。
- `cache:inspect`、`cache:export` 用于缓存排查。

## 12. 扩展约束

新增外部接口时必须同时定义：调用触发条件、缓存键、TTL、失败行为、字段映射、UI 缺失状态和费用影响。新增数据库结构必须通过 migration。新增路由需补充 TypeScript 契约和文档。不得把真实生产域名配置、Token、日志、构建产物或数据库运行目录提交到仓库。
