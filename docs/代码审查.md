# 真实 API 与 SQL 缓存集成审查

## 审查结论

本轮未发现阻断提交的缺陷。当前实现满足三项关键边界：浏览器只调用站内 `/api/v1/home`；第三方凭证仅在后端环境变量中使用；所有第三方响应缓存均持久化到 SQL `api_cache` 表，未使用进程内业务缓存。

## 已验证项

- `npm run typecheck` 通过。
- `npm run build` 通过。
- `GET /api/v1/health` 返回 `status: ok`，本地数据库模式为 `pglite`。
- `GET /api/v1/home` 返回 5 个 Hero、4 个各 6 条的影视分区和 3 条 IMDb 新闻，所有八个来源操作均成功。
- SQL 中已持久化 TMDB、豆瓣、IMDb 缓存记录和 Provider 同步日志，进程重启后仍可命中。
- 首页桌面端真实标题、图片、IMDb 评分、缺失评分状态和新闻均正确渲染，浏览器控制台无错误或警告，页面无横向溢出。
- 源码扫描确认前端仅在 `src/services/home-api.ts` 调用 `/api/v1/home`，没有第三方 API 地址或密钥。

## 剩余风险

- 首次冷缓存需要调用多个上游接口，耗时受第三方服务影响；已将 Hero 富化限制为随机五部并使用 SQL TTL 缓存降低后续请求成本。
- 当前跨源评分采用保守标题与年份匹配，宁可显示“暂无”也不制造错误评分；后续详情页应通过持久化外部 ID 映射提升命中率。
- 当前仅完成首页聚合接口；搜索、探索、详情、登录和管理员后台仍应在后续阶段按同一 Provider 与 SQL 缓存边界实现。
